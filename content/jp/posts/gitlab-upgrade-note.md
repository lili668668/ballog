## TL;DR<br/>
<br/>
GitLab のアップグレードは、ただただ骨の折れる仕事です。私はこの作業を台風による休みの日にやるのが好きです。台風の日は外出できず、同時に誰も出勤していないので、うっかり GitLab をシャットダウンさせてしまっても、プレッシャーがないからです。<br/>
<br/>
このメモは、次の台風休みがやってきて GitLab をアップグレードするときに、これまでにぶつかった凸凹や傷跡をもう一度おさらいできるように書いたものです。<br/>
<br/>
## バックアップ<br/>
<br/>
アップグレードには必ずリスクがあります。システムのアップグレードは得することもあれば損することもあるので、アップグレード前にはきちんとバックアップを取り、Change Log をよく読んでおきましょう。<br/>
<br/>
## GitLab のアップグレードパスを調べる<br/>
<br/>
GitLab のアップグレードは、決まったバージョン番号の順序に沿って上げていけば、データベースの migration がすべて順調に進むことが保証されます。そのため、この<br/>
<br/>
https://gitlab-com.gitlab.io/support/toolbox/upgrade-path<br/>
<br/>
GitLab のアップグレードパス検索サイトは、とても重要です。<br/>
<br/>
このアップグレードパス検索サイトさえあれば、もう成功は半分達成したようなものです！<br/>
<br/>
## アップグレード後の状態確認<br/>
<br/>
状態確認は3項目あります。<br/>
<br/>
1. サイトは生きているか？<br/>
  - 通常アップグレードすると GitLab は再起動し、サイトはしばらく起動中の状態のままになります。なので 502 が見えても、焦らず、慌てず、恐れず。もし朝ごはんを食べて戻ってきてもまだ復旧していなければ、そのとき初めて恐れればよいのです。<br/>
2. `sudo gitlab-rake gitlab:check`<br/>
  - GitLab のその他のコンポーネントの状態です。<br/>
  - このコマンドは GitLab がマシンにベアメタルでインストールされている場合にのみ、そのままコピペして使えます。Docker を使っている場合は、Docker 用のコマンドを探す必要があります。<br/>
3. `sudo gitlab-rake gitlab:doctor:secrets`<br/>
  - GitLab インスタンス内の「暗号化データ」が、現在の secrets 鍵で正しく復号できるかどうかを確認します。<br/>
  - このコマンドは GitLab がマシンにベアメタルでインストールされている場合にのみ、そのままコピペして使えます。Docker を使っている場合は、Docker 用のコマンドを探す必要があります。<br/>
  - このコマンドは1時間ほど動かないまま実行され続けることもあるので、それほど大きなアップグレードでなければ、実は確認しなくても構いません。<br/>
<br/>
## アップグレードのまたアップグレード<br/>
<br/>
たいてい、長い間アップグレードしていなかった場合、アップグレードパスは台風休み1回では終わらないほど長くなります。そういうときは、台風休みが2回必要です！（X）<br/>
<br/>
たいていアップグレードが終わると、すぐに次のアップグレードをやりたくなりますが、次のパスに進む前に、まず1つ重要な画面を開いて確認することをおすすめします。<br/>
<br/>
https://[domain]/admin/background_migrations<br/>
<br/>
この画面は GitLab 内部データベースの migration 進捗ページです。<br/>
<br/>
運が良ければ、この画面はガラガラで何もありません。運が悪ければ、このページには昼ごはんを食べて戻ってきても消化しきれないほどの migration が山ほど並んでいます。<br/>
<br/>
忍耐力を備えたエンジニアとして、通常やるべきことは、この山盛りの migration が消化し終わるのを待ってから次のステップに進むことです。なにしろ、台風休みから戻ってきたら GitLab の内部データベースがぐちゃぐちゃになっている、なんて事態は誰も望んでいませんから。<br/>
<br/>
## 永遠に動かないあの migration<br/>
<br/>
昼ごはんを食べて戻ってきて、うきうきしながら migration ページを開くと、まだ終わっていないのが見えます。終わっていないどころか、`RemoveOldJobTokens: p_ci_builds` が 0.00% のまま止まっています。CKA の試験を受けて戻ってきても、まだ動き出していません。<br/>
<br/>
忍耐力のないエンジニアとして、直感的に何かおかしいと気づくはずです。そこで辿って調べていくと、これは単にシステムがアイドル状態になるのを待ってから動こうとしているだけだと分かります。なんだそれは、台風の日こそあなたが一番暇なときでしょう、システムまで給料泥棒になろうっていうんですか？<br/>
<br/>
そんなときに必要なのが<br/>
<br/>
`sudo gitlab-rake gitlab:background_migrations:finalize[RemoveOldJobTokens,p_ci_builds,id,'[]']`<br/>
<br/>
これで強制実行してやります。<br/>
